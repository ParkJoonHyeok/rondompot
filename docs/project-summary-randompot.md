# RandomPot 프로젝트 설명문 (포트폴리오/면접용)

## 1) 프로젝트 한 줄 소개
RandomPot은 친구/모임/술자리/게임팟에서 빠르게 사용할 수 있는 캐주얼 랜덤 놀이 웹서비스입니다.  
룰렛, 벌칙 뽑기, 랜덤 미션, 팀나누기를 웹에서 즉시 실행할 수 있도록 MVP로 구현하고 배포까지 완료했습니다.

## 2) 개발 목적
- 오프라인 모임에서 "빠르게 랜덤 결정"이 필요할 때 쓸 수 있는 실용 도구를 만들고자 했습니다.
- 복잡한 회원 기능보다 즉시 사용성(입력 -> 실행 -> 공유)을 우선해 사용자 진입 장벽을 낮추는 데 집중했습니다.

## 3) 주요 기능
- **룰렛**
  - 사용자 관점: 입력한 항목 중 하나를 즉시 랜덤 선택
  - 구현 관점: 항목 정제/검증 후 API 호출, 결과/실패 메시지 분기 처리
- **벌칙 뽑기**
  - 사용자 관점: 벌칙 후보에서 빠르게 1개 추첨
  - 구현 관점: 룰렛과 동일한 입력/검증 흐름을 유지하며 기능 목적만 분리
- **랜덤 미션**
  - 사용자 관점: 미션 후보 중 1개를 랜덤 선택
  - 구현 관점: 일관된 API 계약과 UI 상호작용 패턴 유지
- **팀나누기**
  - 사용자 관점: 참가자를 팀 수에 맞춰 랜덤 분배
  - 구현 관점: 팀 수/참가자 수 검증과 팀 결과 렌더링(줄바꿈/가독성) 처리

## 4) 기술 스택
- **Frontend**: Next.js, TypeScript
- **Backend**: Spring Boot, Java
- **Deployment**: Vercel(프론트), Render(백엔드)

## 5) 본인 구현 포인트
- MVP 우선 설계: 핵심 기능 4종을 먼저 안정화하고 배포 기준을 맞춤
- 기능/로직/API 계약 유지 원칙: 개선 작업 시 기존 동작 회귀를 최소화
- 모바일 UX 1차 개선: 입력/버튼/결과 가독성과 터치 사용성 보강
- 공유/복사 기능: 결과 텍스트 복사, Web Share API, 페이지 링크 복사 제공
- sessionStorage 기반 입력 복원: 새로고침/동일 탭 재진입 시 입력값 복원
- 최근 결과 히스토리: 추가/복사/공유/삭제/전체 비우기 구현
- 운영 품질 보강:
  - README, QA 체크리스트, 배포 노트, 공유 링크 설계 노트 문서화
  - metadata/OG, robots/sitemap, not-found/error, manifest 기본 적용

## 6) 기술적 판단 포인트
- **과한 공통화/리팩토링 보류**
  - MVP 단계에서 구조 개선보다 안정적인 동작 유지가 우선이라고 판단
  - 변경 범위를 좁혀 회귀 리스크와 검증 비용을 줄임
- **공유 링크 기능 즉시 구현 보류**
  - URL 직렬화/백엔드 저장형 모두 추가 비용과 운영 리스크가 커서 1차에서는 보류
  - 대신 결과 텍스트 공유 + 페이지 링크 공유로 사용자 목적을 우선 충족
- **page 단위 최소 수정 전략**
  - 공용 컴포넌트 영향 범위를 키우지 않고 기능별로 점진 개선 가능
  - 빠른 검증/되돌리기 용이성 확보

## 7) 트러블슈팅/개선 사례
- 모바일 UX 개선을 공용 컴포넌트 대수정 없이 page 단위 패턴 확장으로 적용
- 결과 복사/공유/세션 복원/히스토리를 백엔드 변경 없이 프론트에서 해결
- sessionStorage 저장 시 동일 값 재기록을 줄여 불필요 write를 줄인 미세 최적화 적용

## 8) 추후 개선 후보 (현재 미구현)
- 공유 링크 고도화(고유 결과 링크)
- 히스토리 UX 개선(조회/필터/정리 방식 보강)
- 모바일 UX 추가 개선(폼/결과 상호작용 디테일 보완)

## 9) 면접용 5줄 요약
RandomPot은 모임에서 빠르게 사용할 수 있는 랜덤 도구 웹서비스로, 룰렛/벌칙/미션/팀나누기 4개 기능을 MVP로 구현하고 배포까지 완료했습니다.  
저는 기능 안정성을 우선해 API 계약과 기존 동작을 유지하면서 모바일 UX, 결과 복사/공유, 페이지 링크 복사, 입력 복원, 히스토리 기능을 단계적으로 보강했습니다.  
또한 metadata/robots/sitemap/not-found/error/manifest 같은 운영 품질 요소를 함께 정리했습니다.  
공유 링크 같은 확장 기능은 즉시 구현하지 않고 비용/리스크를 비교해 보류하는 판단을 했습니다.  
이 프로젝트를 통해 "작게 나누어 빠르게 개선하되 회귀를 통제하는 방식"을 실무적으로 체득했습니다.
