# RandomPot Share Link Design Notes (1st)

## 1) 현재 상태 요약
- 현재 지원 공유 방식
  - 결과 텍스트 복사
  - Web Share API (지원 브라우저)
  - 페이지 링크 복사
- 현재 미지원 범위
  - 결과 상태를 URL에 직렬화해 재현하는 기능
  - 결과 상태를 백엔드에 저장해 고유 링크로 조회하는 기능
- 결론: 지금은 "서비스 진입 링크 공유"와 "결과 텍스트 공유" 단계이며, **결과 재현 링크는 없음**.

---

## 2) 공유 링크 구현 후보안 비교

## A. URL query/hash 기반
예: `?title=...&items=...&result=...` 또는 `#...` 형태로 상태 포함

- 장점
  - 백엔드 변경 없이 빠르게 시도 가능
  - 링크만으로 재현 가능 (이론상)
- 단점
  - URL 길이 제한/인코딩 복잡도
  - 항목 개수가 많으면 링크가 과도하게 길어짐
  - 주소 노출 데이터(제목/항목) 민감도 관리 어려움
  - 파싱 실패/버전 호환 이슈 발생 가능
- 현재 MVP 적합성
  - **낮음~중간**: 구현 자체는 가능하지만, 안정성/품질 리스크 대비 이득이 제한적

## B. session/local storage 기반 간접 공유
예: 로컬 저장 후 같은 브라우저에서만 복원

- 장점
  - 구현 간단
  - 현재 구조(sessionStorage)와 유사
- 단점
  - 타인에게 링크 공유 불가 (기기/브라우저 종속)
  - "고유 공유 링크" 요구를 충족하지 못함
- 현재 MVP 적합성
  - **공유 링크 관점에서는 부적합**

## C. 백엔드 저장형 공유 링크
예: 결과 저장 -> 공유 ID 발급 -> `/share/{id}` 조회

- 장점
  - 짧고 안정적인 링크
  - URL 길이/인코딩 문제 최소화
  - 추후 조회수/만료/삭제 정책 확장 가능
- 단점
  - 백엔드 API/DB/보안 정책 추가 필요
  - 운영 비용과 장애 면적 증가
  - MVP 범위를 넘어가는 작업
- 현재 MVP 적합성
  - **중장기 적합**, 단기(MVP 안정화 단계)에는 무거움

---

## 3) 현재 단계 추천안
- **지금 당장 구현 추천:** 없음
- **보류 권장:** A(URL query/hash), C(백엔드 저장형)
- **추후 후보 우선순위:** C > A

판단 근거 (현재 단계):
- 안정성 우선: 기존 기능(실행/복사/공유/세션/히스토리) 유지가 더 중요
- 개발 비용 대비 효과: "결과 텍스트 공유 + 페이지 링크 복사"로 현재 사용자 목적의 상당 부분 충족
- 배포 리스크: 공유 링크는 파싱/저장/보안 이슈로 회귀 범위가 커짐

---

## 4) 추후 실제 구현 시 최소 요구사항

### 데이터 최소 단위
- 기능 타입(roulette/punishment/mission/team)
- 제목
- 입력 항목 목록
- 결과값(또는 팀 분배 결과)
- 생성 시각(옵션)

### 수정 범위
- 프론트
  - 공유 링크 생성 트리거 UI
  - 링크 진입 시 상태 복원 처리
- 백엔드(C안 선택 시)
  - 공유 생성 API
  - 공유 조회 API
  - 저장소(DB 또는 대체 저장소)

### 최소 체크 포인트
- 보안/악용
  - 과도한 payload/스팸 생성 제한
  - 입력 검증 및 저장 크기 제한
- URL 길이
  - A안은 길이 제한/인코딩 실패 처리 필수
- 개인정보/민감정보
  - 사용자 입력 노출 정책 명확화
- 만료/정리 정책
  - C안은 데이터 보존 기간/정리 기준 필요
